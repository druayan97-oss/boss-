# Boss直评 (Boss Direct Review)

> **职场透明化社区，让招聘方行为更透明。**

**Boss直评** 是一个专注于提升招聘市场信息对称性的职场社区项目。通过用户真实的提名与评价，我们旨在为求职者提供关于招聘方（HR/猎头）行为的透明参考，帮助求职者识别“已读不回”的沟通黑洞，同时表彰那些反馈及时、态度专业的“Nice HR”。

---

## 解决的用户痛点

在当前的互联网招聘环境（以 Boss 直聘为代表）中，求职者面临着严重的**信息不对称**和**情绪内耗**。本项目通过社区驱动的信号机制，精准打击以下四大痛点：

| 痛点维度 | 现象描述 | 心理影响 |
| :--- | :--- | :--- |
| **沟通黑洞** | 简历投递后显示“已读”，但 HR 长期不回复、不拒绝。 | 极度的不确定性与焦虑感。 |
| **筛选盲盒** | 不清楚特定 HR 的筛选偏好（如院校、专业、回复习惯）。 | 盲目投递导致效率低下，挫败感强。 |
| **地位不对等** | 招聘方掌握所有主动权，求职者缺乏反向评价与避雷渠道。 | 职场弱势地位带来的无力感。 |
| **练习缺失** | 缺乏真实的 IM 沟通模拟环境，新手求职者容易在话术上“踩坑”。 | 面对心仪岗位时因紧张或经验不足而错失机会。 |

### 痛点深度解析

#### 沟通黑洞：已读不回的“冷暴力”
**现状**：求职者最痛苦的不是被拒绝，而是被无视。在 Boss 直聘等平台上，大量 HR 仅将平台作为简历收集器，对于不符合要求的候选人往往采取“已读不回”策略。
- **Boss直评的解法**：通过 **“已读不回 HR 榜”** 和 **“简历黑洞公司榜”**，将这种隐形行为显性化。
- **价值**：帮助求职者提前避开那些“只看不招”或“沟通效率极低”的坑位，节省宝贵的时间和精力。

#### 筛选规则的不透明性
**现状**：很多岗位在 JD（职位描述）中写得非常宽泛，但实际筛选时有极强的潜规则（如：只要 985、只要工科、只要有大厂实习）。
- **Boss直评的解法**：在 **HR 详情页** 展示基于真实提名的 **“院校层级”** 和 **“专业门类”** 统计。
- **价值**：求职者可以根据历史数据（如：该 HR 曾回复过 211 工科生，但从未回复过双非文科生）来判断自己的匹配度，实现“精准打击”。

#### 缺乏反馈的“单机”求职体验
**现状**：求职者在投递过程中几乎得不到任何关于自己“招呼语”或“简历初筛”质量的反馈。
- **Boss直评的解法**：**“猜 Boss 练习”** 模块。通过模拟真实的 IM 聊天界面，让用户在模拟局中测试不同背景和招呼语的回复概率。
- **价值**：将枯燥的投递过程转化为“可练习、可反馈”的游戏化体验，提升求职者的沟通技巧和心理韧性。

#### 职场正向激励的缺失
**现状**：虽然吐槽很多，但市场上也存在大量专业、高效、友好的 HR，他们的价值在混乱的招聘市场中被埋没了。
- **Boss直评的解法**：**“Nice HR 红榜”** 和 **“秒回侠榜”**。
- **价值**：通过正向引导，鼓励健康的招聘文化，同时也为求职者指明了那些值得优先投递的“宝藏 HR”。

### 典型用户场景

#### 场景 A：大厂实习生“避雷”
> **用户**：某 211 大三学生，正在寻找暑期实习。
> **痛点**：投递了 50 个岗位，40 个已读不回，不知道是自己简历不行还是 HR 根本不招人。
> **使用 Boss直评**：打开“已读不回榜”，发现其中 10 个 HR 长期霸榜。果断取消关注，转而投递“Nice HR 榜”上的岗位，次日即获得面试邀请。

#### 场景 B：求职新手“话术练习”
> **用户**：应届毕业生，第一次使用招聘软件，不知道怎么打招呼。
> **痛点**：发出的招呼语石沉大海，怀疑自己说话太死板。
> **使用 Boss直评**：进入“猜 Boss 练习”模拟局。通过观察系统生成的不同院校、不同专业的招呼语及其对应的回复结果，悟出了“针对性描述”的重要性，优化了自己的话术。

### 总结

**Boss直评** 不仅仅是一个榜单工具，它是一个**求职者的情报站**。它解决的核心痛点是**招聘过程中的不可预测性**。通过将零散的求职遭遇聚合为结构化的社区信号，它赋予了求职者“知情权”和“选择权”，从而在根本上改善了求职体验。

## 核心功能

项目目前处于前端原型阶段（v1），已实现以下核心业务模块：

| 模块 | 功能描述 | 状态 |
| :--- | :--- | :--- |
| **职聘榜单** | 提供“已读不回 HR 榜”、“简历黑洞公司榜”及“Nice HR 榜”，基于用户提名数据实时排序。 | ✅ 已实现 |
| **HR 详情** | 展示特定招聘方的标签统计、提名记录、岗位 JD 及社区评分。 | ✅ 已实现 |
| **匿名提名** | 支持用户对招聘方进行行为打标（如：秒回、沉默、专业等），支持覆盖更新。 | ✅ 已实现 |
| **猜 Boss 练习** | 模拟真实 IM 聊天场景的互动游戏，帮助求职者通过历史数据预判招聘方的回复概率。 | ✅ 已实现 |
| **个人档案** | 用户可配置自己的招呼语、院校层级与专业门类，用于模拟练习与数据匹配。 | ✅ 已实现 |
| **每周简报** | 社区维度的周度数据汇总与趋势分析入口。 | 🚧 迭代中 |

## 技术栈

本项目前端部分采用现代 Web 开发技术栈，注重性能与开发体验：

- **框架**: [React 19](https://react.dev/) (最新稳定版)
- **构建工具**: [Vite 6](https://vitejs.dev/)
- **语言**: [TypeScript](https://www.typescriptlang.org/)
- **样式**: [TailwindCSS 4](https://tailwindcss.com/) (高性能原子化 CSS)
- **路由**: [React Router 7](https://reactrouter.com/)
- **图标**: [Lucide React](https://lucide.dev/)
- **动画**: [Motion](https://motion.dev/) (原 Framer Motion)
- **测试**: [Vitest](https://vitest.dev/) + [React Testing Library](https://testing-library.com/docs/react-testing-library/intro/)

## 快速开始

### 环境要求

- Node.js 18.x 或更高版本
- pnpm / npm / yarn

### 本地开发

1. **克隆仓库**
   ```bash
   git clone https://github.com/druayan97-oss/boss-.git
   cd boss-
   ```

2. **安装依赖**
   ```bash
   cd boss-zhipin-chat-ui
   npm install
   ```

3. **启动开发服务器**
   ```bash
   npm run dev
   ```
   访问 [http://localhost:3000](http://localhost:3000) 查看应用。

4. **运行测试**
   ```bash
   npm run test
   ```

## 开发者指南

### 原型阶段说明
当前版本为 **前端交互原型**，具有以下特性：
- **本地存储**: 所有用户操作（提名、档案修改、游戏进度）均持久化在浏览器的 `localStorage` 中。
- **模拟数据**: 初始数据由 `src/data/seed.ts` 提供，包含演示用的 HR 档案与提名记录。
- **无后端依赖**: 暂不需要配置 API 密钥或数据库，可直接运行。

所有新增的功能或技术方案应先在 `docs/` 目录下创建对应的 Markdown 文档，并遵循 `YYYY-MM-DD-<主题>.md` 的命名约定。
