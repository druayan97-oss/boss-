# Boss直评 — 产品规格（完整产品与分阶段优先级）

**文档日期**：2026-04-05  
**状态**：共识记录（供后续实现与评审使用）  
**说明**：本文档汇总已对齐的产品决策；目标是交付**完整产品**，通过 **P0 → P1 → P2** 分阶段实现与上线。**P0 并非「唯一终点」**，P1/P2 同属产品范围，仅**优先级较低者后排期**。未拍板事项见 §5。

---

## 1. 背景与要解决的问题

- 求职者在招聘平台（如 Boss 直聘）侧常见痛点：**已读不回**、筛选规则不透明、关系上偏被动。
- 产品设想 **Boss直评**：
  1. **打榜/提名**：由求职者对招聘方（以 HR 为粒度）进行公开记录与榜单，形成可参考的社区信号。
  2. **传播与栏目（P1）**：**每周数据简报**与**分享卡片**，把聚合后的职场信号做成可截图、可转发的站外素材（§3.4）。  
  3. **模拟练习（P2）**：用社区贡献的「招呼文案」与可解释的**行为模拟**，让用户以**代入某 Boss** 的方式练习「回/不回」判断，并量化**与模型真值的一致性**（准确率）。

## 2. 已对齐的产品方向

### 2.1 分阶段优先级（完整产品）


| 阶段     | 优先级              | 范围摘要                                                                           |
| ------ | ---------------- | ------------------------------------------------------------------------------ |
| **P0** | **最高**           | **打榜核心闭环**：账号、提名、多类榜单、公司与 HR 规范化（§2.6）。**邮箱验证整链不在 P0**（见 §2.4、§3.3.13）。**不包含用户侧举报**（见 §2.3、§3.3.11）。 |
| **P1** | **中**            | **全量榜单与话题能力**（§2.8 中标记为 P1 的条目）、**每周数据简报**与**分享卡片**栏目（§3.4）、标签/维度扩展。                               |
| **P2** | **较低（仍属正式产品范围）** | **举报系统**（用户入口 + 核实后处置）+ **模拟招呼 / 猜 Boss 回复行为**（§3.3）+ **邮箱验证**（分阶段强制策略 §2.4，详 §3.3.13）；同属 P2，实现顺序可并行或按依赖编排（如验证可与举报并行、与模拟解耦）。 |


### 2.2 「真实」与展示策略


| 决策         | 内容                                                                              |
| ---------- | ------------------------------------------------------------------------------- |
| **标识方式**   | **不弱化**：榜单侧以**可指向真实场景**的信息为主（如 HR 姓名、公司、招聘岗位等），不做「仅公司/岗位聚合弱化个人」的替代方案（至少当前共识如此）。 |
| **模拟模块表述** | 对外文案须明确为 **社区驱动的模拟练习**，**非** Boss 直聘官方功能、**非**对真实 HR 聊天记录的回放（见 §3.3.2）。         |


### 2.3 内容与审核策略


| 决策     | 内容                                                                           |
| ------ | ---------------------------------------------------------------------------- |
| **凭证** | **不要求用户上传截图/附件**；避免额外存储与基于上传材料的审核链路。                                         |
| **AI** | **不依赖 AI** 做内容审核或凭证分析（成本与复杂度考量）；**模拟模块**亦**不依赖 LLM** 生成真值或招呼语（若未来单独立项需重新评审）。 |
| **治理** | **P0**：**不设用户侧举报入口**；争议与明显问题依赖 **运营后台人工**（若工程上预留「后台隐藏单条提名」能力则优先用后台，非必须）。**P2**：上线 **举报系统**（见 §3.3.11）；用户可对**提名记录**等发起举报，**核实成立后默认处置为「隐藏」**——公开侧不可见、**不参与榜单计数**；数据侧建议 **软隐藏**（保留审计），**不**在首版做自动 HR 降权、**不**默认物理删除、**不**做复杂多分支工单（可迭代）。 |


### 2.4 账号与身份


| 决策                 | 内容                                                                                                                                                                                                                                                                                                  |
| ------------------ | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **登录方式**           | **邮箱登录**；参与提名/打榜等写入类操作需登录（与「非完全匿名」一致，便于频控与追责）。                                                                                                                                                                                                                                                      |
| **邮箱验证（已定稿规则；实现排期 P2）** | **交付**：完整**邮件验证链路**（发信、验证链接、`email_verified` 或同类字段、写入门禁与 §2.4 分阶段策略）在 **P2** 落地（§3.3.13）。**P0** 仅 **邮箱注册/登录与会话**，**不**发验证邮件、**不**按验证状态拦截写入。**以下为验证能力上线后的产品规则（与 P2 同期或验证子版本内启用）**：**长期**：产品需具备**邮件验证链路**，作为正式规则写进数据模型与权限判断。**冷启动**：当平台**累计注册用户数不足 200** 时，**不强制**用户完成「点击邮件验证链接」即可**正常注册并执行写入类操作**（提名/打榜等）。**扩容后**：当**累计注册用户数达到或超过 200**，或运营通过 **功能开关** **提前**开启强制验证时，**强制**未完成邮箱验证的账号**不得**执行写入类操作（或采用等价策略：仅只读浏览），直至用户完成验证。计数口径：**已完成注册流程的独立账号数**（实现时以数据库唯一用户表为准）。 |


### 2.5 提名去重（同用户—同 HR）


| 决策            | 内容                                                                    |
| ------------- | --------------------------------------------------------------------- |
| **一人一票（记录级）** | 同一登录用户对**同一个 HR 实体**在系统中**至多存在一条有效提名记录**；**榜单计数上该用户对该 HR 至多贡献 1 票**。  |
| **变更方式**      | 用户可通过 **编辑** 该条记录更新内容；或通过 **撤销/删除** 该条记录后 **重新提交**（撤销后票数相应收回，直至再次提交）。 |


### 2.6 公司与 HR 名称规范化（无 LLM）


| 维度         | 策略                                                                                                                                                                             |
| ---------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| **公司（高频）** | 维护一份**人工/运营维护的高频公司名单**（canonical 名称 + 可选别名）。用户输入先经**轻量规范化**（见下文化洗「通用步骤」），再与名单做**模糊匹配**（如编辑距离、Jaro–Winkler 等，**阈值与候选条数上限**在实现时调参）；匹配成功则**挂靠 canonical id**，榜单与聚合以 canonical 为准。 |
| **公司（低频）** | 未命中名单的**小公司/低曝光名**：**不做**强归一，原样保留规范化后的字符串即可；**误匹配成本可接受**（产品共识）。后续若某名频次升高，可再**加入名单**。                                                                                           |
| **HR 姓名**  | **不调用 LLM**。采用**可审计的规则管道**生成用于匹配的 **`hr_key`（内部键）**；**展示名**可保留用户原始输入或「清洗后展示名」（实现二选一，需防止过度改写引发争议）。**同用户—同 HR** 的去重（§2.5）在逻辑上基于 **`canonical_company`（或低频下的规范化公司串） + `hr_key`**。 |


**HR 清洗管道——建议实现的规则层（自上而下，均可单测；命中规则应可记录便于排错）：**

1. **Unicode 规范化**：统一为 **NFKC**，消除兼容字形、全角/半角混用带来的假差异。
2. **不可见与空白**：去除零宽字符；全角空格等映射为普通空格；**trim** 与**连续空白折叠**为单空格。
3. **剥离职位/身份噪音（词典 + 边界）**：从**后缀/前缀**移除高频非姓名片段，例如：`HR`、`人事`、`招聘`、`猎头`、`经理`、`主管`、`负责人`、`老师` 等；**须避免**误删过短真名（规则优先匹配**完整词/分隔符边界**，慎用单字规则）。
4. **括号与分隔符段**：对形如 `张三（人事）`、`李四｜招聘`、`王五 - HR` 等，按模式**去掉括号内职位段**或**竖线/短横后的职位段**；保留主体姓名部分。
5. **英文片段**：对拉丁字母部分做 **Unicode case fold** 生成匹配键；**展示**可保留用户原大小写。
6. **重复与异常**：连续重复字符（如 `张张`）可标记为**待二次确认**或按策略折叠为单字（可选，需防误伤）；对**过长/过短/纯符号**输入可拒绝或要求用户修正。
7. **可选增强（仍无 LLM）**：维护小型**同形字/易混字映射表**（用户手写易混字，非模型生成）；对**常见拼音输入法近音错字**可用**预置纠错表**（按批添加，避免泛化误纠）。

**说明**：以上管道解决的是「**同一真人在不同输入法下的写法收敛**」；**「小张」与「张明」是否为同一人**等语义问题**不在规则侧自动解决**；P0 依赖运营后台；P2 举报上线后可对争议条目走 **隐藏**（§3.3.11）。

### 2.7 榜单排序（P0，已定稿）

| 决策 | 内容 |
|------|------|
| **策略** | 采用 **B：计数 + 时间衰减/新近度**——以 **有效提名数**（或产品定义的加权票数）为基础，同时纳入 **时间维度**：**同等票数下，近期仍有新提名或记录更新的 HR/公司更靠前**；亦可对综合分做**温和衰减**，使长期无新互动的实体逐步下移（具体公式、半衰期或 λ 由实现调参，需可配置）。 |
| **「最近活动」口径** | 以指向该 HR 实体（或公司聚合）的 **提名记录 `updated_at` / 新建时间** 为准：用户 **编辑或删后重发** 提名视为活动刷新（与 §2.5 一致）。 |
| **多榜一致** | **个人（HR）榜**与 **公司聚合榜**默认共用同一套排序哲学（计数 + 新近度）；不同榜（如 Nice 与「不回」）若分榜，可在同一框架下用不同计数口径。 |
| **用户说明** | 榜单页提供**一句话解释**（如「综合提名数量与近期更新」），避免被理解为纯人数排名。 |
| **与 P2 举报** | **举报加权惩罚**仍按 §5：待 P2 举报稳定后再评估是否对隐藏/争议记录额外降权。 |

### 2.8 榜单体系与话题策略（完整产品；已定方向）

以下榜单 **均可做**（与头脑风暴共识一致）：底层尽量共用 **提名 + 标签 + 聚合** 数据模型，**不同「榜」可为同一数据的多种呈现皮肤或筛选器**；排序默认遵循 **§2.7**。**梗向名称**用于传播时，须在页内保留 **统计口径说明**，避免被误解为官方定性或人身攻击工具。

#### 2.8.1 核心向（求职信号、长期价值）

| 榜单 / 视图 | 说明 | 建议阶段 |
|-------------|------|----------|
| **已读不回 / 沉默 HR 榜** | 与产品初衷最直接对应；按标签与提名聚合到 HR 实体。 | **P0 起做** |

| **「人间蒸发区」/「已读不回名人堂」** | 同一类数据的栏目化、月度/周榜包装；可轮换标题防审美疲劳。 | **P1** |

## 3. 完整产品功能范围与交付优先级

### 3.1 P0：打榜与治理（优先交付）

- **用户体系**：邮箱注册/登录、会话管理。**邮件验证**（发信、`email_verified`、200 用户阈值与运营开关、未验证禁止写入）在 **P2** 交付（§2.4、§3.3.13）；P0 不实现验证链路，不按验证状态限制写入。  
- **提名/记录**：用户可提交 **HR 可识别信息 + 公司 + 岗位**（字段细则技术定稿）。**同一用户对同一 HR 仅一条有效记录**，可编辑或删后重发（§2.5）。公司走 **§2.6** 名单与模糊匹配；HR 名走规则 **`hr_key`**（无 LLM）。**P2 模拟上线后**：每次提名保存/更新时，同步写入 **§3.3.9** 所述 **`hr_greeting_candidate`**（招呼语 + 院校层级 + 门类来自当时 **设置页档案**）；P0 可只落提名、**不建模拟素材表**直至 P2。  
- **榜单（P0 最小集）**：**已读不回/沉默 HR**、**简历黑洞公司**、**Nice HR** 等（与 §2.8.1 一致）；**排序**遵循 **§2.7**。完整梗向榜、活动皮肤见 **§2.8**，主要在 **P1** 铺开。  
- **举报**：**不在 P0 交付**；与 **P2** 一并规划（§3.3.11）。

### 3.2 P1：全量榜单、标签扩展与话题运营（核心稳定后）

- **榜单全集**：按 **§2.8** 落地其余榜单与「皮肤」；**「听不懂人话 HR」** 等与标签/理由字段挂钩的维度，以**标签体系统一挂接**，避免每条榜一套独立逻辑。  
- **数据依赖**：岗位聚类、行业视图、梗榜所需字段在 P1 迭代中逐步补齐。  
- **合规**：梗向对外素材默认 **偏统计与脱敏**；强 UGC 语录类（§2.8.2）与 **§3.4 分享卡片** 默认 **不含可识别个人** 或二级页才展示详情（产品规则单独立项）。

### 3.3 P2（较低优先级）：模拟招呼 / 猜 Boss 回复准确率

本节为**完整产品**的正式功能说明，实现可再拆 **P2.1 / P2.2** 子里程碑，但**不属于「可选玩具」**，而是与主产品共享账号与 HR 实体体系。

#### 3.3.1 目标与体验概述

- **目标**：让求职者通过**高频短回合**练习，体会「不同 Boss 风格下，自己是否误判对方会回还是不会回」；用**可解释的分数**反馈学习感。  
- **核心幻想**：用户 **随机或自选代入一名真实存在于打榜中的 HR**（`hr_id` 与榜单同源）；以 **Boss 视角** 审阅一条条 **真实求职者招呼**——这些招呼来自 **曾提名过该 HR 的用户**：他们在提名时所保存的 **招呼语文本 + 院校层级 + 专业门类（粗粒度）** 会作为「这条消息的背景卡片」展示（**不展示提名用户身份**）。用户对每条消息决策：**回复** 或 **不回复**（首版 **二态**）。  
- **结算**：一局若干回合后，对比系统给出的 **「标准答案」序列**，计算 **准确率**（正确回合数 / 总回合数），并可展示历史曲线、与全站均值对比等（实现分阶段）。

#### 3.3.2 边界与非承诺（合规与预期管理）

- **非官方**：与 Boss 直聘等**无 API 合作**假设；界面与文案避免「同步真实 Boss」「真实聊天记录回放」等表述。  
- **真值性质**：标准答案来自 **平台内可观测信号 + 公开规则** 合成的 **启发式模型**，是对社区标签/统计的**游戏化投影**，**不保证**与任何具体真人的真实行为一致。  
- **用户生成内容**：招呼与 **粗粒度背景标签** 来自提名用户；**禁止展示具体学校全名、手机号等**；仅允许 §3.3.3 所列 **层级 + 门类** 与招呼正文。需遵守 §2.3 治理；P2 举报上线后对违规内容可走 **隐藏**。

#### 3.3.3 用户侧数据：设置页「求职者档案」（供提名与模拟池共用）

每个登录用户在 **设置页** 维护一套 **粗粒度** 背景（**不写清具体学校名**，降低隐私与审核风险；后续若加「硕士/博士」等学历层次可单列枚举，首版可选）：


| 字段 | 说明 |
|------|------|
| **招呼语** | 用户自定义的 **Boss 打招呼模板**（自我介绍/开场白）；**提名某 HR 时**与下方背景一并写入该 HR 的 **模拟收件箱素材**（见 §3.3.5、§3.3.12）。 |
| **院校层级** | **单选枚举**（可配置顺序与文案）：例如 **清北 / Top3**、**C9（非清北口径时可与 Top3 互斥或合并，产品定）**、**211**、**双非一本**、**二本**、**专科 / 高职**、**海外院校（粗）**、**不便透露 / 其他**。 |
| **专业门类** | **单选枚举**：**文科**、**商科**、**理科**、**工科**、**交叉 / 其他**。 |


**与提名的关系（已定方向）**：

- 用户 **提名某个真实 HR** 时，系统使用 **当时设置页中的招呼语 + 院校层级 + 专业门类** 生成（或更新）一条 **绑定 `hr_id` 的素材记录**（逻辑上可与提名记录 **1:1** 或合并存储，见 §3.3.12 C）。  
- **同一用户对同一 HR 仅一条有效提名**（§2.5）：用户 **编辑提名** 或 **改设置后再保存提名** 时，**刷新**对应素材快照，使模拟局看到的始终是「该用户对该 Boss 的最新版本」——若需保留历史版本可后置，首版 **覆盖即可**。  

**用途**：

1. **按 HR 分池的模拟收件箱**：仅 **提名过该 `hr_id`** 的用户贡献素材；玩家代入该 Boss 时看到的招呼 **来自真实提名者**，带 **脱敏后的背景标签**（层级 + 门类），**不暴露用户 id、昵称、具体学校**。  
2. **个性化（P2.2+ 可选）**：在「同一 `hr_id` 池内」再按门类/层级加权抽样等，非首版必须。

#### 3.3.4 Boss 选择与「带入」

- **来源**：必须从 **已存在于打榜体系的 HR 实体** 中选择（或 **随机代入**），以便挂载 **统计特征**（见 3.3.6）且 **收件箱素材池** 非空（至少 **有他人提名过** 才更有「真实招呼」，仍不足 K 则走预置兜底）。  
- **随机模式**：从「满足最低数据量阈值」的 `hr_id` 集合中均匀抽样（阈值见 §3.3.12 H）。  
- **指定模式**：用户在排行榜点选某一 HR；若数据不足，提示换人或改随机。

#### 3.3.5 招呼流（收件箱）如何生成

- **分池规则（核心）**：收件箱素材 **按 `hr_id` 隔离**——玩家代入 Boss `H` 时，只从 **「曾向 `H` 提交过有效提名」的用户** 所产生的记录中抽取 **招呼语 + 院校层级 + 专业门类** 快照。  
- **内容来源**：  
  1. **主来源**：上述 **提名绑定素材**（与 §3.3.3 一致）；  
  2. **兜底**：若该 `hr_id` 下可用条数 **不足 K 条**，用 **运营预置模板**（可带虚构但中性的「背景标签」）补足，并在结算或脚注说明「部分为示例文案」。  
- **展示形态（类 IM）**：每轮展示 **招呼正文** + **小字/标签展示**「院校层级 · 专业门类」（不对照到真人）；用户判断 **回 / 不回**。  
- **抽取策略（首版建议）**：在同一 `hr_id` 池内 **均匀随机** 或 **低曝光优先**；**排除当前玩家本人** 贡献的条目（避免自己审自己）。同一文案在一局内重复是否允许——产品定（建议 **每局每快照最多 1 次**）。  
- **节奏**：单局 **固定回合数 K**（如 10～20）或 **限时**（二选一）；**局间**可休息，防疲劳。  
- **与举报/隐藏**：提名记录被 **隐藏**（§3.3.11）时，对应素材 **不得再进入** 抽取池。

#### 3.3.6 「标准答案」与准确率（无 LLM）

因无法获取真实 HR 私聊记录，**真值**必须由 **平台已有数据** 推导，且规则需**可对用户简述**（透明度越高，信任成本越低）。

**推荐默认方案（可实现、可解释）：基于该 HR 实体的社区统计先验**

1. 从打榜数据聚合该 HR 的 **行为特征**，例如：
  - **不回倾向** `p_no_reply`：与「已读不回/不回」相关提名权重 / 总有效反馈数（权重方案技术定稿）。  
  - **正向倾向** `p_nice`：Nice HR 提名占比（若有）。
2. 将 `p_no_reply` 约束在 `[p_min, p_max]` 并做 **平滑**（避免新实体极端 0/1），得到 **单条消息的标准答案采样概率**：
  - 对每条展示的招呼，**独立**（或弱相关马尔可夫——后置）按伯努利抽样：`不回` 概率 ≈ `f(p_no_reply, 本局难度系数)`。
3. **用户操作**：用户选 **回** 或 **不回**；与抽样结果一致则本回合 **正确**。
4. **准确率**：`正确数 / K`。

**可选增强（仍无 LLM，P2.2+）**：

- **招呼长度/关键词微调**：短招呼略调高「被回」概率等——**须谨慎**，避免引入歧视性规则。  
- **众包真值**：同一 Boss 多名玩家决策聚合后更新先验（复杂，易博弈，**不放在首版**）。

**与用户心理的衔接**：结算页用一句话解释，例如：「本局标准答案来自该 Boss 在社区中的**历史标签统计**，用于练习而非真实对话预测。」  

#### 3.3.7 计分、排行与复玩

- **个人**：保存**最近 N 局**准确率、总回合数；可选 **按 Boss 维度** 的最佳/最差记录。  
- **轻量排行（可选）**：全站「模拟连胜/均分」榜——**低优先级**，且需防刷（频控、每 Boss 每日局数上限）。  
- **复玩**：同一 Boss 可重复开局；随机种子不同导致序列不同，**准确率**为统计意义下的练习结果。

#### 3.3.8 内容与风控（P2 必做底线）

- **招呼池**：长度限制、基本敏感词/广告拦截（规则引擎，非 LLM）；P2 举报核实后对条目 **隐藏**。  
- **滥用**：同账号高频刷局、脚本行为用 **频控** 与 §4 账号策略处理。  
- **版权与抄袭**：P2 举报隐藏 + 简单重复检测（SimHash 等可选）。

#### 3.3.9 与 P0 数据模型的关系

- **HR 实体 id** 与打榜共用；模拟局 **引用** `hr_id`。  
- **求职者档案**（设置页）：`job_seeker_profile` 存 **招呼语、院校层级枚举、专业门类枚举**；**提名记录** 与 **按 `hr_id` 分池的素材** 通过 `nomination_id` 或联合唯一键 `(user_id, hr_id)` 关联（实现二选一，须与 §2.5 一致）。  
- **招呼素材**：逻辑表 **`hr_greeting_candidate`**（或等价命名）至少含 `hr_id`、`greeting_text_snapshot`、`school_tier`、`field_track`、`source_nomination_id`、`status`；**前端永不展示** `source_user_id`。  
- **局（session）**、**回合（turn）**、**用户选择**、**系统真值** 落库；`simulation_turn` 快照含 **招呼正文 + 院校层级 + 门类**（见 §3.3.12），便于复盘与客诉。

#### 3.3.10 实现分期建议（在同一 P2 优先级内的先后）


| 子阶段      | 内容                                                                        |
| -------- | ------------------------------------------------------------------------- |
| **P2.1** | **设置页**求职者档案（**院校层级 / 专业门类 / 招呼语**）+ **提名→按 hr_id 素材池** + 随机/自选代入真实 HR + 二态回合 + **统计先验真值** + 准确率展示。 |
| **P2.2** | 防刷与局数限制、历史曲线、按 Boss 统计、可选全站轻榜、脱敏与重复检测增强。                                  |

#### 3.3.11 举报系统（P2；P0 不做）

- **范围**：**与模拟模块同属 P2**；**P0 不提供** App/站内举报入口。  
- **对象（首版建议）**：**提名记录**为主；招呼池 UGC 可在同一套举报模型下扩展（同一「可举报实体」抽象）。  
- **流程（简化）**：用户提交举报（理由枚举 + 短文本）→ **运营/审核核实**（首版人工即可）→ 处置。  
- **核实成立后的默认处置**：**仅隐藏**——该条在**公开列表、详情、榜单计数**中均视为不存在；数据库层 **软删除/visibility=hidden**，便于审计与误操作恢复。  
- **明确不做（首版）**：不因举报自动 **HR 降权**、不默认 **物理删除**、不自动封禁提交者（若需封禁走独立账号策略）。  
- **与打榜关系**：隐藏后该提名对应的票数从 HR/公司聚合中剔除；是否通知提交者、申诉流程 **后续迭代**。  

#### 3.3.12 详细设计：代入 HR、猜「回 / 不回」

本节在 §3.3.1～3.3.9 之上给出 **可对接工程** 的交互、状态、数据与真值管线；具体字段名以实现为准。

##### A. 用户旅程（屏幕流）

1. **入口**：主导航 **「模拟」** Tab；或从 **榜单 HR 详情** 的「代入 Ta 练一局」进入（预填 `hr_id`）。  
2. **设置页 / 档案（强烈建议首次引导，可跳过开局但影响「能否贡献素材」）**：编辑 **招呼语**、**院校层级**、**专业门类**（§3.3.3）。**未完善档案的用户**仍可玩模拟，但若从未提名过当前 Boss，则 **不会向该 HR 池贡献素材**；开局前可提示「去设置页完善，提名后你的招呼会出现在别人的模拟里」。  
3. **选 Boss**：  
   - **指定**：从可玩列表选一名 HR（列表数据来自打榜实体 + §3.3.4 数据量门槛）。  
   - **随机**：服务端在「满足门槛的 `hr_id` 集合」中均匀抽样；若集合为空，提示「数据不足，请稍后再试」。  
4. **局中（类 IM）**：共 **K 轮**。每轮展示 **招呼正文** + **院校层级 / 专业门类标签**（来自 **该 `hr_id` 提名用户快照**，§3.3.5），用户点 **「会回复」** 或 **「不回复」**（二选一）。  
5. **单轮反馈（首版建议）**：**不**在每轮立即揭示对错，避免用户反推随机规律；**P2.2** 可选开启「即时反馈」开关。  
6. **结算页**：展示 **准确率**、**K**、本局快照的 **先验说明**（见 D 与 G）；按钮 **同一 Boss 再来**、**换 Boss**、**回榜单**。  

##### B. 客户端 / 会话状态机

| 状态 | 含义 |
|------|------|
| `idle` | 未开局。 |
| `boss_select` | 待选或待确认 Boss（若深链已带 `hr_id` 可跳过）。 |
| `playing` | 进行中；`current_turn ∈ [0, K-1]`。 |
| `completed` | 已结算；结果已持久化。 |
| `abandoned` | 用户退出或超时未续玩。 |

**弃局策略（P2.1 建议）**：超过 **T_abandon**（如 24h）未完成的会话标记为 `abandoned`，**不参与**个人历史胜率统计；是否落库半局可选（建议 **仍落库** 便于分析流失）。  

##### C. 逻辑数据模型（表级草案）

- **`job_seeker_profile`**：`user_id`（PK）、`greeting_text`、`school_tier`（枚举：清北/Top3、C9、211、双非一本、二本、专科/高职、海外粗类、不便透露等）、`field_track`（枚举：文/商/理/工/交叉）、`updated_at`。  
- **`hr_greeting_candidate`（按 Boss 分池）**：`id`、`hr_id`、`nomination_id`（或 `user_id+hr_id` 唯一）、`greeting_text_snapshot`、`school_tier_snapshot`、`field_track_snapshot`、`status`（`active`/`hidden`）、`updated_at`；**内部**存 `source_user_id` 供风控与举报，**API 不返回**。运营预置兜底可 `source_user_id` 空、`nomination_id` 空。  
- （弃用全局泛池命名 **`greeting_pool_entry`** 的「与 hr 无关」模式；若代码保留表名，须 **强制非空 `hr_id`**。）  
- **`simulation_session`**：`id`、`user_id`、`hr_id`、`boss_select_mode`（`pick` \| `random`）、`k_rounds`、`rng_seed`（字符串或大整数，创建时生成）、`p_no_reply_effective`（**开局快照**， DECIMAL，便于审计与客诉解释）、`state`、`started_at`、`completed_at`。  
- **`simulation_turn`**：`id`、`session_id`、`turn_index`（0-based）、`greeting_text_snapshot`、`school_tier_snapshot`、`field_track_snapshot`（或 `hr_greeting_candidate_id` + 冗余三字段）、`ground_truth_is_no_reply`、`user_chose_no_reply`、`is_correct`、`created_at`。  
- **`hr_behavior_stats`（物化表或视图）**：`hr_id`、`w_no_reply`、`w_positive`、`w_neutral`、`n_total`、各权重定义与提名标签映射在实现期定稿；供开局计算 `p_no_reply_effective` 时读取 **快照**（会话创建时刻复制进 `simulation_session` 亦可）。  

##### D. 真值管线（无 LLM，首版默认算法）

**目标**：对每一局生成 **K 个独立的** `ground_truth_is_no_reply`，数学上等价于已知 `p_eff` 的 **K 次伯努利试验**；**可复现**（同 `rng_seed` + 同参数可重放）。

1. **从提名聚合原始倾向**（示例，与标签体系绑定）：  
   - `w_no_reply`：带「已读不回 / 不回」类标签的提名加权之和；  
   - `w_positive`：Nice 等正向标签加权之和；  
   - `w_neutral`：其余有效提名权重；  
   - `n_total = w_no_reply + w_positive + w_neutral`（或简化为条数计数，**二选一在实现期锁定**）。  
2. **平滑（拉普拉斯型，防小样本极端）**：  
   - `p_smooth = (w_no_reply + α) / (n_total + α·β)`，建议 **α=2**，**β=3**（对应三类伪计数，可调）。  
3. **截断**：`p_eff = clamp(p_smooth, p_min, p_max)`，建议 **p_min=0.2，p_max=0.8**（可调，避免全对/全错局过多）。  
4. **极低数据（可选）**：若 `n_total < N_min_play`，拒绝开局（§3.3.4）或 **与 0.5 线性混合**：`p_eff ← λ·p_eff + (1-λ)·0.5`，`λ = n_total / N_min_play`（仅当产品允许弱数据也玩时）。  
5. **逐轮抽样（默认）**：用 **确定性伪随机** 流（如 `seed = hash(session_id, rng_seed, turn_index)` 或标准 PRNG 顺序取数），对每个 `turn_index` 采样 `u ~ Uniform(0,1)`，**`ground_truth_is_no_reply = (u < p_eff)`**。  
6. **判分**：用户选「不回复」⇔ `user_chose_no_reply=true`；与 `ground_truth_is_no_reply` **相等则 `is_correct=true`**。  
7. **准确率**：`sum(is_correct) / K`。  

**可选 B（分层抽样，P2.2+）**：预先固定本局恰好 `round(K·p_eff)` 个「不回」回合，再随机打乱顺序——方差更小、体验更「像人设」；首版用 **独立伯努利** 即可。  

##### E. 招呼与背景抽取（与真值独立）

- **作用域**：仅查询 **`hr_id = 本局 Boss`** 且 `status=active` 的 **`hr_greeting_candidate`**。  
- **排除**：`source_user_id == 当前玩家` 的条目（必排）；可选再排除与当前玩家同 `school_tier` 等（P2.2+）。  
- **顺序**：池内 **均匀随机** 或 **低曝光优先**（按进入各 `simulation_turn` 次数升序）。  
- **快照**：每轮将 **招呼正文 + school_tier + field_track** 写入 `simulation_turn`，避免后续改档改提名引发争议。  
- **冷启动**：该 `hr_id` 下条目不足 K → **运营预置模板**（带中性标签）补足，并在 UI/结算脚注标明。  

##### F. API 草图（REST 风格，可按前端习惯合并）

- `POST /simulation/sessions`：body `{ hr_id?: string, random?: boolean, k?: number }` → 创建会话、预生成 K 轮 `ground_truth` 与 **招呼+背景标签** 快照；返回 `session_id`、首条 payload（`greeting_text`, `school_tier`, `field_track`）。  
- `GET /simulation/sessions/:id`：返回状态、当前 `turn_index`、当前轮 **正文 + 标签**（若 playing）。  
- `POST /simulation/sessions/:id/answers`：body `{ turn_index, choice: "reply" | "no_reply" }` → 写 `user_chose_no_reply`、`is_correct`；若未满局返回下一招呼；若满局返回结算摘要。  
- `GET /simulation/sessions/:id/summary`：结算后返回准确率、`p_no_reply_effective`、文案说明模板。  

（实现可选用 **单一长轮询 / WebSocket** 优化，非必须。）

##### G. 对用户可见的解释文案（固定模板 + 填参）

> 「本局标准答案由该 Boss 在平台上的 **提名标签统计** 经平滑后得到「不回」概率约 **{p_no_reply_effective_fmt}**，再按该概率随机模拟每条消息的结果。**仅供练习与讨论，不代表真实聊天行为。**」

##### H. 防刷与默认参数（与 §5 T4 对齐，可调）

| 参数 | 建议默认值 | 说明 |
|------|------------|------|
| `K` | **12** | 单局回合数。 |
| `N_min_play`（随机 Boss） | **3** | 该 HR 至少 **n_total** 条有效提名方可进入随机池。 |
| `α`（拉普拉斯平滑） | **2** | 见 D.2。 |
| `p_min`, `p_max` | **0.2 / 0.8** | 见 D.3。 |

#### 3.3.13 邮箱验证（P2）

- **范围**：与 §2.4「邮箱验证（已定稿规则；实现排期 P2）」一致；**P0 不交付**本能力。  
- **能力**：发送验证邮件（可点击链接或等价流程）、持久化 **邮箱是否已验证**、后台或配置中心维护 **强制验证开关**；按 **累计注册用户数** 与开关组合应用 **冷启动不强制 / 扩容或开关后强制** 的写入门禁（与 §2.4 表内规则一致）。  
- **与打榜关系**：强制策略生效后，未完成验证的账号不得执行提名等写入；仅只读浏览等策略与 §2.4 等价表述一致。  
- **实现顺序**：可与 **举报**、**模拟** 并行；仅需与用户表、会话与权限中间件对齐，**不阻塞** P0 打榜闭环（P0 阶段无验证门禁）。  

### 3.4 P1：「每周数据简报」栏目与分享卡片（传播）

- **产品形态**：站内独立 **栏目/Tab**（名称可如「本周简报」「职场信号」），与榜单列表并列，便于用户**固定预期**（如每周五更新）与回访。  
- **简报内容**：基于当周（或滚动 7 日）**聚合数据**自动生成 + 运营可改**梗向标题**（仍须与脚注统计口径一致），例如：  
  - 全站/分行业 **新增提名数、活跃公司数**；  
  - **上升最快**的「简历黑洞」公司 / **沉默 HR**（§2.7 排序下的 delta）；  
  - **Nice / 秒回** 正向亮点 1～2 条；  
  - 可选 **「本周迷思」** 一句 meme 文案（非针对具体个人）。  
- **分享卡片**：用户一键生成 **图片卡片**（或标准 OG 预览链接），便于发小红书、微博、朋友圈等。  
  - **默认策略**：卡片主视觉以 **数字 + 公司级/行业级结论 + 产品品牌** 为主，**不默认展示具体 HR 姓名**；若用户选择「分享某榜截图式卡片」，须带 **免责声明**（社区统计、非官方）。  
  - **技术**：服务端拼图（如 headless 渲染）或端上 Canvas 导出均可，以实现成本为准。  
- **与 §2.8 联动**：简报可每周轮换 **不同梗榜皮肤** 的「本周之星」模块，强化话题性。  
- **阶段**：与 P1 榜单扩展同期或略晚于 P0 最小榜（先保证有稳定数据再推简报）。

## 4. 风险与实现侧隐含要求（开发需知情）

以下不构成法律意见，仅作产品/工程风险提示：

- **可识别个人信息 + 负面评价** 易引发名誉与合规争议；**无上传凭证** 会提高 **误伤、造谣、商业抹黑** 概率。**P0 无用户举报**时，运营响应与后台能力若不足，争议易外溢至舆情；**P2 举报**上线后应以 **隐藏处置** 与审核 SLA 兜底。  
- **防刷**：在「无附件、无 AI」前提下，更依赖 **账号维度频控**、**重复条目合并**、**同 HR 多来源合并策略**、必要时 **设备/IP 辅助风控**（具体方案待技术选型）。  
- **与第三方平台关系**：产品不假设与 Boss 直聘等存在官方合作；对外表述需避免暗示「官方数据」或「实时同步对方行为」——**模拟模块尤甚**（§3.3.2）。  
- **P2 特有风险**：招呼池 **UGC** 可能含骚扰、广告、隐私泄露；**真值算法**若不透明易被质疑「操控分数」——需 **规则文档化** 与 **用户可见的简短说明**。  
- **传播功能（§3.4）**：简报与分享卡片会 **放大** 榜单与梗文案的站外传播，须严格执行 **默认脱敏** 与免责声明，避免成为针对个人的「大字报」素材；舆情与法务风险随 DAU 上升而上升。
