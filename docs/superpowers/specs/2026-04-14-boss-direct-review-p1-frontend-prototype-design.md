# Boss直评 P1 前端原型设计 Spec

**日期**: 2026-04-14  
**状态**: 已确认，待实现规划  
**关联文档**:
- `docs/2026-04-05-boss-direct-review-design.md`
- `docs/2026-04-09-boss-direct-review-frontend-prd.md`
- `boss-zhipin-chat-ui/` 现有视觉原型

## 1. 目标

基于现有 `boss-zhipin-chat-ui` 的视觉语言，重做一套与产品文档对齐的 `P1` Web 前端原型。该原型不是展示板，而是后续正式前端开发的母版，需要同时满足以下要求：

- 保留现有原型中用户认可的组件气质、色调、卡片质感和聊天页代入感。
- 页面结构、信息架构、流程状态必须对齐产品与前端 PRD。
- 原型覆盖 `P1` 范围，且为多页面、可点击、流程级交互。
- 后续正式前端默认沿用这套原型的页面组织、组件边界和视觉系统。

## 2. 关键决策

### 2.1 交付范围

- 本轮交付范围为 `P1` 原型，而不是仅做 `P0`。
- 原型必须覆盖：
  - 榜单首页
  - HR 详情页
  - 提名页
  - 猜 Boss 落地页
  - 猜 Boss 游玩页
  - 结算页
  - 我的档案 / 设置页
- 页面之间必须具备可点击跳转和可演示流程。

### 2.2 原型类型

- 采用多页面原型，不采用单页展示板。
- 采用流程级可点击交互，不停留在静态状态切换。
- 页面内的关键动作需要真实修改前端本地状态，而不是只弹出假提示。

### 2.3 继承策略

- 对现有 `boss-zhipin-chat-ui` 采用“中继承”策略。
- 保留内容：
  - 清透偏青绿的色调气质
  - 雾面渐变背景与顶部柔光带
  - 白卡片 + 轻阴影 + 较高信息密度的界面感觉
  - 左列表 / 中内容 / 右信息面板的聊天布局节奏
- 重做内容：
  - 页面信息架构
  - 导航结构
  - 组件语义
  - 与 Boss 官网过于接近的文案和视觉暗示

### 2.4 猜 Boss 入口修正

- 原 PRD 中的“双卡入口”仍然保留。
- 但当前产品最新决策为：两个入口暂时都由模拟数据驱动，不承诺真实数据局。
- 双卡的产品定义改为：
  - `随机 Boss 风格模拟`
  - `指定 Boss 风格模拟`
- 两个入口共享同一套游玩页组件，只是开局路径和 Boss 选择方式不同。

## 3. 设计原则

### 3.1 总体方向

这套原型采用“双层视觉语言”：

- `数据产品壳`: 服务于榜单、详情、提名、设置、结算等页面，强调数据可读性和结构秩序。
- `沉浸游玩壳`: 服务于猜 Boss 落地页和游玩页，保留招聘方消息场景的代入感。

两套页面不是两个产品，而是同一个品牌系统下的两种布局模式。统一项包括：

- 品牌色与辅助色
- 卡片风格
- Tag / Badge 体系
- 按钮体系
- 圆角、阴影、留白尺度
- 背景渐变与高光氛围

### 3.2 与 Boss 直聘的关系

- 视觉参考 Boss 招聘方后台的节奏和信息密度，但不做官网式复刻。
- 不使用 Boss 官方导航文案、官方功能话术、官方感过强的互动细节。
- 页面必须明确为 `Boss直评` 自有产品，而不是第三方页面伪装。

### 3.3 产品气质

- 专业、克制、可信。
- 梗感通过榜单名、说明文案和运营皮肤表达，不通过花哨 UI 表达。
- 数据与说明必须可见，尤其是排序口径、练习性质、非官方说明。

## 4. 信息架构

### 4.1 共享外壳

所有页面共用顶层品牌导航，包含：

- `榜单`
- `猜 Boss`
- `本周简报`
- `我的档案`
- 登录态区域

要求：

- 顶部导航固定为 `Boss直评` 自有品牌头部。
- 保留现有原型里的清透光感，但去掉 Boss 官网式导航项。
- 使用统一的页面背景、页宽和卡片容器系统。

### 4.2 页面清单

#### 1. 榜单首页 / 榜单页

默认落在 `已读不回 HR 榜`，具备榜单切换能力。

结构：

- 顶部榜单 tab
- 左侧榜单主列表
- 右侧说明和行动入口

职责：

- 承接产品首页
- 展示当前最核心的社区信号
- 引导进入 HR 详情和提名流程

#### 2. HR 详情页

结构：

- 返回入口
- HR / 公司 / 岗位信息
- 标签与统计信息
- `练一局` 与 `提名 / 更新提名`
- 脱敏提名摘要列表

职责：

- 承接榜单点击后的详情查看
- 提供进入提名和指定 Boss 模拟的深链

#### 3. 提名页

结构：

- 单栏聚焦表单
- HR 姓名、公司、岗位、JD、标签
- 覆盖更新提示
- 登录门禁反馈

职责：

- 完成提名写入流程
- 明确一人一票与覆盖逻辑

#### 4. 猜 Boss 落地页

结构：

- 页面说明
- 两张等权入口卡
- Boss 风格选择或预填提示

双卡定义：

- `随机 Boss 风格模拟`
- `指定 Boss 风格模拟`

职责：

- 明确本玩法当前为模拟练习
- 区分两类开局路径

#### 5. 猜 Boss 游玩页

结构：

- 左栏：待处理招呼列表
- 中栏：消息区 + 二选一操作
- 右栏：Boss 卡 + 岗位 + JD

职责：

- 承担最强沉浸感
- 复用现有原型中最成熟的聊天页视觉语言
- 通过多轮状态推进形成可玩的练习流程

#### 6. 结算页

结构：

- 准确率核心视觉
- 本局 Boss 信息
- 模式与练习说明
- 先验说明
- 二次行动按钮

职责：

- 收束游玩体验
- 输出练习反馈
- 给出后续路径

#### 7. 我的档案 / 设置页

结构：

- 招呼语
- 院校层级
- 专业门类
- 使用说明

职责：

- 固定用户侧练习档案
- 为未来真实素材逻辑预留稳定交互模型

### 4.3 全局浮层

需要统一设计并在整套原型中复用：

- 登录弹窗
- Toast
- 确认弹窗

其中：

- 登录弹窗只拦写入类操作
- 游玩中断确认弹窗只在进行中的会话里出现

## 5. 视觉系统

### 5.1 色彩职责

采用“功能主色 + 氛围色”的分层方式：

- `功能主色`: Boss 绿，用于主按钮、选中态、关键数字、进度条。
- `氛围色`: 偏青绿的 mint / teal，用于背景高光、hover 光感、品牌气质点缀。
- `模拟标识色`: 橙色，用于练习素材和模拟模式提示。
- `负向色`: 红色，仅用于错误、负向标签、未读提醒。

原则：

- 不做全页面大面积纯绿色。
- 不把青绿色直接当成功能态颜色，避免语义混乱。
- 所有颜色必须服从信息表达，而不是纯装饰。

### 5.2 页面外壳

#### 数据产品壳

用于：

- 榜单
- HR 详情
- 提名
- 设置
- 结算

特征：

- 浅雾面渐变背景
- 顶部柔光带
- 白色卡片为主
- 信息清晰、结构稳定

#### 沉浸游玩壳

用于：

- 猜 Boss 落地页
- 猜 Boss 游玩页

特征：

- 延续现有原型的浅青灰背景和顶部光感
- 保持高密度消息界面节奏
- 强化沉浸感，但不伪装官方产品

### 5.3 核心组件

应优先抽出并复用以下组件：

- `ShellHeader`
- `PanelCard`
- `Badge`
- `ActionBar`
- `InfoStat`
- `RankRow`
- `SessionItem`
- `MessageBubble`
- `BossMetaCard`
- `FormSection`
- `Modal`
- `Toast`

组件边界要求：

- 同一组件必须能跨多个页面复用。
- 不允许把整个页面写成一个难拆分的大型视图。
- 原型阶段即保持组件语义稳定，方便后续开发直接沿用。

## 6. 页面规格

### 6.1 榜单首页 / 榜单页

必须包含：

- 榜单 tab 切换
- 榜单排序说明
- 榜单条目卡
- 右侧提名 CTA
- 数据简报预告位

榜单条目至少展示：

- 排名
- HR 或公司主体信息
- 岗位
- 标签
- 提名数
- 最近更新时间
- 详情 / 提名入口

约束：

- 榜单页必须看起来像数据产品，而不是消息页。
- 低样本条目需要单独的占位态。
- 榜单排序说明必须常驻可见。

### 6.2 HR 详情页

必须包含：

- HR 展示名
- 公司名
- 岗位
- 标签统计
- 最近活动时间
- `练一局`
- `提名 / 更新提名`
- 提名摘要列表

约束：

- 提名列表只显示脱敏信息。
- 从此页进入 `指定 Boss 风格模拟` 时可预填 `hr_id`，但仍先经过双卡入口。

### 6.3 提名页

字段：

- HR 姓名
- 公司名
- 招聘岗位
- 职位描述（JD）
- 标签

必须支持：

- 未登录拦截
- 已提名覆盖提示
- 成功提交后回流详情或榜单

约束：

- 提交动作必须真的更新本地状态。
- 成功提交后应同步影响榜单统计与详情展示。

### 6.4 猜 Boss 落地页

必须包含：

- 玩法说明
- 非官方练习说明
- 两张模式卡

双卡文案必须明确区分：

- `随机 Boss 风格模拟`: 直接开局，系统随机选择风格对象。
- `指定 Boss 风格模拟`: 先选或预填一个具体 Boss 风格，再进入练习。

约束：

- 两张卡不能只是文案微调，必须在开局路径上真的不同。
- 两张卡都应明确说明当前为模拟练习，而不是使用真实聊天记录。

### 6.5 猜 Boss 游玩页

三栏结构固定：

- 左侧：招呼列表
- 中间：消息正文与操作
- 右侧：Boss 信息卡

每轮操作只有两个：

- `会回复`
- `不回复`

交互要求：

- 选项提交后锁定当前轮次
- 自动推进下一轮
- 左侧列表状态同步变化
- 已答轮次可回看不可改

右侧 Boss 卡展示：

- HR 展示名
- 公司
- 岗位
- JD 摘要或正文
- 若资料不足，显示“部分信息为示例文案”

### 6.6 结算页

必须包含：

- 准确率
- 答对轮数 / 总轮数
- 本局 Boss 信息
- 当前模式说明
- 一句话先验说明
- `再来一局`
- `换个 Boss`
- `回榜单`

约束：

- 结算页必须对“练习性质”做清晰说明。
- 准确率是视觉中心，其他信息退后。

### 6.7 我的档案 / 设置页

字段固定为：

- 招呼语
- 院校层级
- 专业门类

说明文案需要强调：

- 档案用于练习素材和玩法背景。
- 不展示用户真实身份。

约束：

- 保存动作必须修改本地状态。
- 字段命名和枚举尽量对齐主 PRD，为后续接后端保留稳定结构。

## 7. 交互与状态模型

### 7.1 状态分组

前端原型统一使用本地状态管理，至少拆分为：

- `auth`
- `boards`
- `nominations`
- `profile`
- `play`
- `ui`

要求：

- 页面共享同一份数据源，不允许每页各自维护一套死数据。
- 关键状态落到 `localStorage`，保证刷新后原型流程仍可继续。

### 7.2 关键流程

#### 流程 1: 榜单 -> HR 详情 -> 提名

- 从榜单进入 HR 详情。
- 从 HR 详情进入提名页。
- 未登录时弹登录弹窗。
- 登录完成后回到原动作。
- 若已提名，提交时执行覆盖更新。
- 提交成功后同步刷新：
  - 榜单计数
  - 最近更新时间
  - HR 详情统计

#### 流程 2: 猜 Boss 落地 -> 模式选择 -> 游玩 -> 结算

- 落地页先选择一种模拟模式。
- `随机 Boss 风格模拟` 直接随机开局。
- `指定 Boss 风格模拟` 先选 Boss 风格或接受预填对象后开局。
- 游玩页进行多轮选择。
- 结算页输出准确率和解释。

#### 流程 3: 档案更新 -> 后续流程使用

- 设置页保存档案后立即更新全局 profile 状态。
- 提名页和游玩页读取同一套档案信息。
- 未来接真实数据时，这套 UI 不需要重做。

## 8. 假数据与演示规则

### 8.1 假数据原则

- 整套原型共用单一假数据源。
- 假数据必须支持流程联动。
- 所有提交成功都需要真实修改前端状态，而不是只展示静态成功态。

### 8.2 最小假数据集

建议至少包含：

- 3 个榜单
- 8 到 10 个 HR
- 2 到 3 家高频公司
- 12 到 16 条提名记录
- 2 组模拟招呼模板
- 至少 1 个随机模拟样本会话
- 至少 1 个指定 Boss 样本会话

### 8.3 当前猜 Boss 数据策略

- 当前两个模式都用模拟招呼数据。
- 随机模式从 Boss 风格池随机取样。
- 指定模式围绕一个具体 Boss 风格生成消息与信息卡。
- 所有模式都应给出“练习素材 / 非真实聊天记录”提示。

## 9. 边界状态

原型必须覆盖以下边界状态：

- 榜单低样本态
- 榜单空状态
- 登录门禁态
- 已提名覆盖态
- 指定 Boss 资料不足态
- 游玩中断确认态
- 结算解释态

这些状态不是附属页面，而是正式原型的一部分。

## 10. 明确不做

本轮原型不包含：

- 真实数据池接入
- 举报系统
- 邮箱验证链路
- 周报完整内容页
- 分享卡片导出功能
- 复杂筛选组合器
- 后端 API
- 真实服务端持久化

移动端要求仅为响应式可用，不做精细打磨版本。

## 11. 验收标准

实现完成后，原型应满足：

- 能从榜单完整走到详情、提名、返回刷新。
- 能从榜单或详情进入猜 Boss，并完成一个完整会话。
- 双卡模式差异清晰，且都不误导为真实数据。
- 榜单页与游玩页视觉统一，但布局职责清楚。
- 现有原型的审美气质被保留，且不再和产品文档冲突。
- 后续开发可直接按当前页面结构、组件边界和状态字段继续实现。
