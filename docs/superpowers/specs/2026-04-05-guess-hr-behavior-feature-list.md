# 猜 Boss 行为 — 功能与交互清单

**说明**：本文档只描述 **「猜 Boss 行为」** 的 **功能与交互**，不涉及后端实现。完整产品定义见 [2026-04-05-boss-direct-review-design.md](./2026-04-05-boss-direct-review-design.md)。

---

## 一、这是什么产品（简要）

**Boss直评**面向求职者：大家可以在平台上记录、讨论招聘方（以 HR 为主）的求职体验（例如是否已读不回等），并形成榜单与话题，让求职信息更透明一些。

其中 **「猜 Boss 行为」** 是一个 **练习型玩法**：你 **假装自己是某位真实存在于本平台榜单里的招聘方（HR）**，在 **仿 Boss 直聘招聘方看到的沟通界面** 里，处理一条条 **求职者发来的招呼**。你要判断：**如果你是这位 HR，你会回复这条消息，还是不回复？** 一局结束后，系统会告诉你猜对了多少轮，并说明标准答案大致依据什么规则生成——**仅供娱乐与练习，不代表 Boss 直聘官方，也不等于真实聊天记录。**

## 二、功能与交互清单（猜 Boss 行为）

### 2.1 入口：两张卡片

- [ ] 进入本玩法时，首屏 **仅两张卡片**（或两个等大入口模块），**无额外 Tab 作为主入口**：  
  - **卡片一**：明确告知 **本局使用模拟数据**（招呼等为系统生成的练习内容，非真实用户提名）。  
  - **卡片二**：明确告知 **本局使用真实数据**（招呼等来自提名过该 Boss 的用户贡献，不展示对方身份）。  
- [ ] 用户 **点击其中一张** 即选定本局 **数据来源**，再进入后续流程（选 Boss 等）。  
- [ ] **可选增强**：从 **榜单 / HR 详情**「练一局」进入时，可 **预选这位 Boss**，但仍 **先展示两张卡片**（或默认高亮与场景更匹配的一张），由用户确认数据来源后再进入 **与平时完全相同的游玩主界面**。

### 2.2 选 Boss（卡片之后、进主界面前）

- [ ] **自己选**：从列表里选一位 Boss（数据来自本平台榜单；不足时提示换人）。  
- [ ] **随机一位**：系统随机分配一位数据达标的 Boss。  
- [ ] 与 **2.1** 的组合顺序：**先点卡片定数据源 → 再选 Boss**（若深链已带 Boss，可合并步骤，但 **数据源仍须由用户通过双卡之一确认**）。

### 2.3 设置页：我的求职画像（与提名共用）

- [ ] 用户可在 **设置** 里维护：招呼语、院校层级（粗粒度）、专业门类。  
- [ ] **不要求填写具体学校全称**。  
- [ ] 提示：**提名某位 HR 时**会带上以上内容，供 **真实数据** 卡片路径下的其他玩家看到（不暴露你是谁）。

### 2.4 与「提名」的配合（真实数据卡片路径）

- [ ] 提名某位 HR 时使用当前设置；每人每 Boss **一条有效提名**；更新提名或设置后，他人 **真实数据** 局内看到的是 **最新版**。  
- [ ] 提名侧建议可填写 **岗位 / JD 相关正文**（或结构化岗位描述），用于游玩界面右侧 **Boss 卡片** 里的 **职位 JD** 展示（若无长文本，可先只展示岗位名称 + 简短说明，产品可迭代）。

### 2.5 统一游玩主界面（两种卡片进入后 **布局完全一致**）

整体 **参考 Boss 直聘里招聘方（HR）查看求职者消息** 的页面结构（**视觉与信息架构参考**，非官方、无账号打通）。

- [ ] **顶部 — 求职者基本信息**  
  - 展示当前正在处理的这一位「对面求职者」的 **基本信息区**：如院校层级、专业门类、招呼摘要/关键词等（与设置枚举一致）；**不展示**对方真实昵称或具体校名（可用「求职者」或匿名编号）。  

- [ ] **中区 — 主消息区**  
  - 呈现 **对方发来的招呼消息**（气泡或列表，与 IM 一致）。  
  - **当前待判断的一轮**在视觉上明确（如滚动定位、高亮）。  
  - 主操作：**「回复」** / **「不回复」**（二选一）。  

- [ ] **左侧 — 通知 / 未读招呼栏**  
  - **模拟**招聘方常见的 **左侧会话或「未读招呼」列表**：列出本局内 **待处理的招呼条目**（可与 K 轮一一对应或聚合展示），带 **未读** 等状态样式；点击可切换到对应一轮（交互以实现为准，须与轮次一致）。  

- [ ] **主消息区旁 — Boss 小卡片（信息面板）**  
  - 固定或随布局贴在消息区域 **一侧**，展示 **你正在代入的这位 Boss** 的 **信息面板**：  
    - **HR 名字**（与榜单一致展示策略）  
    - **公司名称**  
    - **职位信息**：岗位名称 + **该职位的 JD（职位描述正文）**（内容来自该 Boss 关联的提名/岗位数据或运营配置；无长 JD 时展示岗位名与占位文案）  

- [ ] **真实数据 / 模拟数据** 两种路径下，**上述四块区域布局与组件相同**；仅在 **文案提示**（如顶栏角标「本局为模拟数据」）或 **数据来源说明** 上区分。  

- [ ] **首轮建议**不在每一轮立刻公布对错；**结算页**统一展示结果（可选后续做「每轮即时反馈」）。

### 2.6 结算页

- [ ] 准确率、总轮数、本局是 **模拟数据** 还是 **真实数据**。  
- [ ] 固定说明：标准答案依据该 Boss 在本平台的 **标签/统计** 模拟生成，**非真实聊天预测**。  
- [ ] 操作：**再来一局**（可回到 **两张卡片** 或保留上次卡片选择）、**换 Boss**、**返回榜单**。

### 2.7 体验与限制（用户能感知到的）

- [ ] 设置/提名内容违规或过长时的 **提示**。  
- [ ] 每日局数等限制的 **Toast / 弹窗**。

### 2.8 后续可增强的交互（可选）

- [ ] 真实招呼旁的 **举报**、个人战绩、每轮即时对错开关、混局玩法、轻量排行等。

---

## 三、对用户说明的边界（文案层面）

- [ ] 全链路避免让人以为 **与 Boss 直聘官方实时同步** 或 **在回放真实 App 内记录**。  
- [ ] **模拟数据** 与 **真实用户提名** 在 **入口卡片** 与 **局内角标** 上都要能区分。  
- [ ] **真实数据** 路径仍遵守：不出现具体校全名、电话等敏感信息（见主规格）。
